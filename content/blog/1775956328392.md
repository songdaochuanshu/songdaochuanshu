conhost.exe 无法显示 U+2717
=======================

我想在 Windows 的传统控制台（conhost）中显示 U+2717（✗，BALLOT X）。选中系统默认的等宽字体 Consolas，输出的却是一个带有方框的问号。我在使用 NeoVim 时遇到了这个符号，其还需要 Nerd Fonts ，但甚至支持 Nerd Fonts 的字体很多都不支持这个字符。

![](https://img2024.cnblogs.com/blog/3489664/202604/3489664-20260405180151072-1775690056.png)

在 Stack Overflow 上也有人遇到了同样的问题：

[https://stackoverflow.com/questions/27519142/is-it-possible-to-output-and-display-u2717-ballot-x-in-the-windows-console](https://stackoverflow.com/questions/27519142/is-it-possible-to-output-and-display-u2717-ballot-x-in-the-windows-console)

问题的原因显而易见。 Consolas 不包含这个字形，换一个包含它的字体就好。查阅 fileformat.info 的字形支持列表后发现，DejaVu Sans Mono、Inconsolata 等开源等宽字体确实包含 U+2717 。

[https://www.fileformat.info/info/unicode/char/2717/fontsupport.htm](https://www.fileformat.info/info/unicode/char/2717/fontsupport.htm)

在 Windows 10 较新版本中，conhost 已经可以直接读取已安装的等宽字体，无需像早期版本那样手动编辑注册表。

[https://superuser.com/questions/1767614/possible-to-enable-additional-fonts-in-command-prompt-without-registry-edits](https://superuser.com/questions/1767614/possible-to-enable-additional-fonts-in-command-prompt-without-registry-edits)

在安卓系统中， U+2717 可以正常显示。安卓默认使用的是 Roboto 。但 Roboto 并没有 U+2717 。安卓系统使用了字体回退机制。

[https://www.fileformat.info/info/unicode/font/roboto/list.htm](https://www.fileformat.info/info/unicode/font/roboto/list.htm)

Consolas 里面绝没有中文字符。

![](https://img2024.cnblogs.com/blog/3489664/202604/3489664-20260405180151071-1649517365.png)

把字体设为 Consolas 时，输入中文却能正常显示。

![](https://img2024.cnblogs.com/blog/3489664/202604/3489664-20260405180151090-3759005.png)

这说明 conhost 并非只用单一字体渲染，也存在某种回退机制。按照这个逻辑，既然中文能回退到微软雅黑显示，U+2717 理应也能回退到包含它的系统字体（如 Segoe UI）来显示才对。

那为什么 U+2717 显示不出来？

conhost 的回退机制基于 GDI 的字体链接（Font Link），具体配置位于注册表的 `HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\FontLink\SystemLink` 中。

[https://www.cnblogs.com/nick357/p/3619327.html](https://www.cnblogs.com/nick357/p/3619327.html)

这个键值定义了每个字体的回退链条。但是，这里根本没有 Consolas 。

……这就很奇怪了，那中文是怎么显示的呢？

GDI 处理缺失字形时，实际上有两套机制，它们是叠加关系：

1.  **字体链接（Font Linking）**：由注册表 `SystemLink` 配置，用户可修改。
2.  **字体回退（Font Fallback）**：由渲染引擎（Uniscribe）内置，不需要任何注册表配置。

微软在 Microsoft Learn 官方文档 “Customize font selection with font fallback and font linking” 中解释了这两者的区别：

> Font linking is a mechanism enabled within GDI and takes priority over font fallback. Unlike font fallback, in which the selected font is internally replaced by a predefined font, using font linking it’s possible to link one or more fonts (called “linked fonts”) to another font (called the “base font”). Once you link fonts, you can use the linked fonts to display code points that don’t exist in the base font but do exist in one of the linked fonts. The registry key HKEY\_LOCAL\_MACHINE\\SOFTWARE\\Microsoft\\Windows NT\\CurrentVersion\\FontLink\\SystemLink specifies several composite fonts and their corresponding font files.

[https://learn.microsoft.com/en-us/globalization/fonts-layout/fonts](https://learn.microsoft.com/en-us/globalization/fonts-layout/fonts)

GDI 在 `SystemLink` 中找不到对应条目时，还会用 Uniscribe 查询其内置回退表。中文能正常显示，就是因为 CJK 在 Uniscribe 的内置回退表中有对应的条目。

Uniscribe 的内置回退表是按 Unicode 脚本区块进行粗粒度映射的，主要覆盖各语言的书写系统（CJK、阿拉伯文、泰文等）。而 U+2717（✗）属于 Dingbats 区块（U+2700–U+27BF），其 Unicode 脚本属性为 Common（Zyyy），通用类别为 So（Other Symbol）。这类“不属于任何特定文字书写系统”的符号字符，根本不在 Uniscribe 回退表的覆盖范围内。

微软控制台团队在 2018 年的官方博客中也承认了这一点：

> Console currently uses GDI for text rendering, but GDI doesn't adequately support font-fallback – a mechanism to dynamically find and load an alternative font that contains a glyph missing from the current font.

[https://devblogs.microsoft.com/commandline/windows-command-line-inside-the-windows-console/](https://devblogs.microsoft.com/commandline/windows-command-line-inside-the-windows-console/)

有人向微软提交过几乎一模一样的需求——在 Consolas 中添加 CLI 常用的 Unicode 符号（包括 ✔、✖、★、▶、⚠ 等），这些字符与 U+2717 属于同一区域，面临完全相同的问题。

微软对此高度重视，并将其标记为 Won't Fix 。

> A major point of moving to the Terminal and to the DX renderer is to not fix/update the GDI one anymore. So while it is always possible to improve it, it would be out of scope for what we're trying to achieve.

[https://github.com/microsoft/terminal/issues/387](https://github.com/microsoft/terminal/issues/387)

不过，微软在 2021 年通过 PR #10478 做了一个架构层面的修复：将 conhost 的 GDI 渲染从 `PolyTextOutW` 切换为 `ExtTextOutW`。

[https://github.com/microsoft/terminal/pull/10478](https://github.com/microsoft/terminal/pull/10478)

miniksa 在 PR 中解释：`PolyTextOutW` 的代码中**没有调用 Uniscribe**，所以字形替换根本不会发生；而 `ExtTextOutW` 会先将文本发送给 Uniscribe 进行语言处理，Uniscribe 完成后将调用拆分为带 `ETO_IGNORELANGUAGE` 标志的 `ExtTextOutW` 进行渲染。

这个修复打开了 Uniscribe 字形替换的通道，但它解决的是通道是否通畅的问题，而不是回退表是否完善的问题。U+2717 不在 Uniscribe 回退表中，通道再通畅也无济于事。

虽然这个问题理论上可以通过手动添加字体链接解决，但我不想折腾了。我最终还是放弃了使用老旧的 conhost 。
