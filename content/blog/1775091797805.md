OpenAI 官方出手：把 Codex 接进 Claude Code
==================================

codex-plugin-cc 直接把 Codex 接进 Claude Code，让你在原来的工作流里就能调用 Codex 做代码审查，或者把某个任务直接丢给 Codex 处理

最近 OpenAI 新推出了 `codex-plugin-cc` 。第一感觉就一句话：这东西挺实用。它不是再搞一个新 IDE，也不是让你在几个工具之间来回切，而是直接把 Codex 接进 Claude Code，让你在原来的工作流里就能调用 Codex 做代码审查，或者把某个任务直接丢给 Codex 处理。官方也给了常用的命令：`/codex:review`、`/codex:adversarial-review`、`/codex:rescue`、`/codex:status`、`/codex:result`、`/codex:cancel`。

说白了，这个插件解决的是一个很实际的问题：平时你可能一直在 Claude Code 里写代码，但到发版前、排查 bug 时，或者想让另一个模型给第二视角的时候，总会想切到 Codex。`codex-plugin-cc` 干的就是把这一步省掉。它不是让你放弃 Claude Code，而是让你在 Claude Code 里顺手把 Codex 拉进来。

这玩意儿能干什么？
---------

先看最常用的三件事。

第一，普通代码审查。`/codex:review` 是只读 review，不会改代码，适合看当前未提交改动，也适合拿当前分支去对比 `main` 之类的基线分支。官方里还特别提到，多文件改动的 review 往往比较慢，通常建议放后台跑。

第二，对抗式审查。`/codex:adversarial-review` 不只是找低级 bug，而是专门去质疑实现方向、设计取舍和隐藏假设。比如你刚改了缓存策略、重试逻辑、回滚链路、权限判断，这时候让它来“挑刺”，往往比普通 review 更有价值。

第三，任务移交。`/codex:rescue` 可以直接把问题甩给 Codex，让它去查 bug、试修复、续接上一次任务，或者换个更小的模型跑一遍更快的版本。它支持 `--background`、`--wait`、`--resume`、`--fresh`，还支持指定模型和 effort。

安装前先看两个前提
---------

装这个插件之前，先确认两件事。

一是 电脑上 Node.js 版本要在 18.18 及以上。

二是 Claude Code 版本。Claude Code 官方文档写明，`/plugin` 相关能力需要 1.0.33 或更高版本；如果你执行 `/plugin` 提示未知命令，第一步先跑 `claude --version`，别急着怀疑插件本身。

安装步骤，照着敲就行
----------

这个插件的安装流程不复杂：

    /plugin marketplace add openai/codex-plugin-cc
    /plugin install codex@openai-codex
    /reload-plugins
    /codex:setup
    

`/codex:setup` 会检查你本地的 Codex 是否已经安装、是否已经登录；如果 Codex 没装而且本地有 npm，它还能直接给你安装提示。

如果你电脑上还没装 Codex，再补一条：

    npm install -g @openai/codex
    

如果已经装了，但还没登录：

    !codex login
    

安装完成后，你应该能在 Claude Code 里看到一组 `/codex:` 命令，以及 `/agents` 下面的 `codex:codex-rescue` subagent。

实践
--

如果你是第一次装，不用一上来就把所有命令都试一遍，先从最常见的两条开始。

### 1）提交前跑一遍 review

    /codex:review --background
    /codex:status
    /codex:result
    

思路很简单：先让 Codex 在后台帮你过一遍代码，然后再用 `status` 看进度，用 `result` 看最终输出。对于改动比较多、文件比较散的需求，这种方式比前台硬等舒服很多。

### 2）有争议的改动，直接上 adversarial-review

    /codex:adversarial-review --base main challenge whether this was the right caching and retry design
    

这条命令特别适合发版前那种“代码未必错，但方案不一定稳”的场景。比如权限收敛、数据库迁移、回滚策略、并发处理、可靠性兜底，这些地方普通 review 往往只看写法，对抗式 review 才更容易把坑挖出来。

### 3）卡住了，就把问题交给 Codex

    /codex:rescue investigate why the tests started failing
    /codex:rescue fix the failing test with the smallest safe patch
    /codex:rescue --background investigate the regression
    

`rescue` 这条命令挺像“接力棒”。你在 Claude Code 里卡住了，不想继续自己兜，就直接让 Codex 去查

后台任务这块，设计得还算完整
--------------

这个插件不是那种“命令执行完了就没后文”的路子。它把后台任务单独做成了完整闭环：

    /codex:status
    /codex:result
    /codex:cancel
    

`/codex:status` 可以看当前仓库里正在跑和最近跑完的 Codex 任务；`/codex:result` 会展示某个已完成任务的最终输出；如果有 session ID，还能继续用 `codex resume <session-id>` 回到 Codex 里接着跑；`/codex:cancel` 则是取消仍在执行的后台任务。这个设计很适合“先丢过去跑，继续写别的”的开发节奏。

还有个 review gate，建议别默认常开
-----------------------

`/codex:setup` 还有一个可选项是 review gate：

    /codex:setup --enable-review-gate
    /codex:setup --disable-review-gate
    

官方的说明是：打开以后，插件会在 Claude 准备结束当前响应时，通过 `Stop` hook 触发一次有针对性的 Codex review；如果这轮 review 发现问题，就阻止 stop，让 Claude 继续处理。官方也顺手给了警告：这个机制可能让 Claude 和 Codex 形成长循环，额度消耗会很快，只有在你准备盯着会话看的时候才建议开启。

建议别把它当默认配置。只有在高风险改动上再开，比如数据库迁移、权限改造、支付状态流、删除链路、并发逻辑。普通业务开发没必要给自己加这一层摩擦。

结尾
--

如果你本来就常用 Claude Code，而且已经有“写完想让另一个模型再看一眼”的习惯，那这个插件是值得装的。它最大的价值不是命令多，而是把“Claude 写主流程，Codex 做复查和接力”这件事做顺了。你不用切上下文，不用换工具，不用重开一套会话，直接在原来的工作流里插一段 Codex 能力进去。

但如果你本身就主要用 Codex，或者你根本没有“多模型交叉检查”的需求，那它也不是必装项。说到底，它更像一个工作流增强插件，而不是一个非装不可的底层依赖。
