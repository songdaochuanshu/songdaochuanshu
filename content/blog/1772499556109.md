NVIDIA GPU 集合通信库 NCCL 基本概念简述
============================

本文来自博客园，作者：[T-BARBARIANS](https://www.cnblogs.com/t-bar) **原创**，博文严禁转载，转载必究！

本篇内容主要对 NVIDIA GPU 集合通信库 NCCL 的基本概念做一些肤浅的介绍。文章将从 NCCL 的产生背景，NCCL 是什么，NCCL 初始化关键能力简介，NCCL 支持的通信原语，NCCL 的一些基础技术概念和优化点，等多个角度去阐述。

GPU集群集合通信面临的问题
==============

高效的分布式深度学习训练需要大量 GPU 卡协同工作。在缺乏通信协同工具的场景下，大量数据在多个 GPU 之间的互相传递可能会导致如下问题：

*   数据传输延迟高，网络带宽利用率低，模型训练性能差
*   数据的传输占用了模型训练的绝大部分时间，GPU 集群更多时间处于数据等待状态，GPU 利用率低下，形成资源浪费
*   不同的训练框架都要开发自己的 GPU 通信逻辑，重复劳动、兼容性差、效率低下

面对高性能、低时延集合通信的迫切需求，GPU 集合通信库 NCCL 应运而生。

NCCL是什么
=======

NCCL (NVIDIA Collective Communications Library)，是 NVIDIA 专门为多节点、多 GPU 场景设计的集合通信库，是一个多 GPU 之间的通信框架。

NCCL 源码链接：[https://github.com/NVIDIA/nccl](https://github.com/NVIDIA/nccl)

简单总结起来，NCCL 具备如下能力:

*   能够自动探测所在节点的 PCIe 设备，包括 NIC，PCIe Switch，CPU，GPU，NVSwitch 等
*   自主计算以上 PCIe 设备相互之间的连接关系
*   自主计算以上任意 PCIe 设备之间的最优可达路径（低延时，高带宽）
*   自主预计算 Ring，Tree 等通信算法的逻辑通信通道，并依据实际情况自动选择最优通信算法。并且，在物理通道带宽基础上，以多通道的形式尽可能的用尽硬件资源
*   在单机，多机通信场景下，通过 Ring，Tree 等通信算法，自主实现 GPU 集群之间的拓扑连接，形成集合通信的实际通信链路
*   GPU 之间通信高效并低延迟。单机内支持基于 PCIe、NVLink、NVSwitch 的通信传输方式，多机间支持基于 InfiniBand、ROCE、TCP 的传输方式
*   支持分布式深度学习集合通信 / 点对点通信原语，以 API 的形式暴露给上层应用方，例如集合操作：ncclAllReduce、ncclBroadcast、ncclReduce、ncclAllGather、ncclReduceScatter，点对点操作：ncclSend、ncclRecv

NCCL 通过在集群 GPU 之间自主建立通信拓扑，并通过 API 屏蔽底层复杂业务逻辑的方式，可以很方便的被上层分布式深度学习框架集成，使得上层应用方便的实现分布式深度学习。当前，NCCL 已经是众多分布式深度学习框架的默认集成组件。

生成的 XML 拓扑反映了从 CPU 往下，直到 GPU 或者 NIC 的层次关系。NCCL 正是基于这种层次关系去建立 PCIe 设备间的逻辑连接关系。

PCIe设备间建图
---------

PCIe 设备之间的物理链路关系是事实存在的。但是 NCCL 并不知道，NCCL 需要将这些关系映射到自身的数据结构中，为最终的 channel 计算打基础。

NCCL 通过解析前面生成的 XML 拓扑，从 CPU 开始逐渐往下迭代，直至到达底层硬件 GPU / NIC，再通过回溯的方式：

*   建立路径上 PCIe 设备之间的连接关系
*   记录 PCIe 设备之间彼此连接的带宽信息
*   当前，任意 PCIe 设备都只有自身周边一跳 PCIe 设备的连接关系

直到 XML 中的所有 PCIe 设备都具有基于物理事实的两两连接关系。

计算PCIe设备间的最优可达路径
----------------

基于 PCIe 设备间的两两连接关系，NCCL 会使用 BFS 算法，计算任意 PCIe 设备之间的最优可达路径。例如：推导从 GPU0 出发，其它 PCIe 设备反向到达该 GPU0 的路径，并记录路径类型、路径带宽、路径跳数。

NCCL支持的通信原语
===========

NCCL 支持分布式深度学习所需的核心集合通信原语，为分布式应用提供了多种通信原语的通信接口。

Broadcast
---------

Broadcast 操作将来自某个 rank 的数据复制发送到其他所有 rank。

Scatter
-------

Scatter 操作将某个 rank 的缓冲区按 rank 个数均分为 N 份（N 为 rank 的个数），向每个 rank 发送一份均分后的对应数据。

Gather
------

将多个 rank 上的数据收集到单个 rank 上，Gather 可以理解为是反向的 Scatter 操作。

AllGather
---------

AllGather 将收集所有节点的数据到所有节点上。可以理解为 All Gather = Gather + Broadcast。

AlltoAll
--------

*   每个 rank 将自身数据划分为 N 块（N 为 rank 的个数），其中第 j 块发送给 rank j
*   每个 rank 接收 N 块数据（N 为 rank 的个数），其中第 i 块来自 rank i

也就是说，每个 rank 都会向所有其它 rank 发送不同的数据，同时接收来自所有其他 rank 的不同数据。

Reduce
------

Reduce 将把多个 rank 的数据规约运算到一个 rank 上。

例如：只有 root rank 接收到所有 rank 数据的规约结果。常见 reduce 操作：sum、min、max等。

ReduceScatter
-------------

ReduceScatter 操作与 Reduce 类似，但规约后的结果会被平均分块，然后按 rank 索引分发给各个 rank，每个 rank 只接收到结果的一部分。

AllReduce
---------

在所有的节点上都应用同样的 Reduce 操作，也就是说每个 rank 最终都会得到相同的规约结果。

AllReduce 操作有两种实现方法：

*   Reduce + Broadcast
*   ReduceScatter + AllGather

NCCL通信协议简介
==========

NCCL 当前支持三种协议，分别是 Simple、LL(Low Latency)、LL128(Low Latency, 128-byte chunks)。 每种协议都有自己的优缺点和适用场景，基本特性可以被总结为如下图所示：

Simple
------

Simple 协议的设计目标就是最大化带宽利用率，适用于大消息传输，可以接近硬件峰值带宽。

*   Simple 协议采用内存屏障的方式确保数据的正确排序，导致数据接收端必须等待完整数据块传输完毕后才可访问数据。
*   内存屏障会造成较大的时延开销，对传输小消息尤为明显。

因此，Simple 适合大块数据传输场景，例如 AllReduce 操作，或者对时延不太敏感的场景。

**Simple 特点：传输大块数据，高带宽，高延迟**

LL(Low Latency)
---------------

针对 Simple 协议的高延迟，NCCL 设计了面向小消息的 LL(Low Latency) 协议，目标就是数据传输低延迟。

LL 不再使用内存屏障的方式做数据同步，而是采用标志位的方法进行数据同步。传输数据时伴随发送一个微型标志位以标识数据有效性，接收方可以立即识别数据就绪状态，从而避免了高时延的内存屏障操作。每次只传输 4 字节数据 + 4 字节 flag，共 8 字节。

但是 LL 协议要求待发送数据必须存放于主机内存中，是通过 CPU 去高效轮询，判断数据标志位是否满足发送条件。这样设计完全是为了追求更低的传送时延，将传送数据放置于主机内存的原因可总结为：

*   如果待发送数据位于 GPU 中，那么 CPU 将通过 PCIe 链路去访问 GPU，会造成高时延，通常 > 1μs
*   如果待发送数据位于 host memory，CPU 通过访问 DRAM，可以更快的访问到传送数据以及发送标志位，时延一般在 100ns 左右

但是这样做的代价就是：不能使用 GPU Direct RDMA（Remote Direct Memory Access，一种高性能技术，NIC 可直接读写 GPU 显存，无需 CPU 或主机内存中转），只能通过 CPU copy 或传统 DMA，无法发挥 InfiniBand/RoCE 网卡的高带宽能力。

**LL 特点：传输小消息，低带宽，低延迟**

LL128(Low Latency, 128-byte chunks)
-----------------------------------

Simple 高带宽，高延迟；LL 低带宽，低延迟。因此 NCCL 设计了 LL128 协议，在数据传输时尽可能达到较高带宽，较低延迟的目的。

*   与 LL 类似，采用标志位以标识数据有效性，120 字节数据 + 8 字节 flag，共 128 字节
*   发送端 GPU 会聚合较大数据块后才向 CPU 发出传输就绪通知

数据标识与 CPU 轮询实现低延时，较大数据块实现较高带宽。因此，LL128 具有 **传输较大消息，较高带宽，较低延迟** 的特点。

NCCL数据传输方式
==========

NCCL 对节点内通信，节点间通信，采用了差异化的数据传输策略。

节点内通信
-----

节点内通信，NCCL 主要有 GPUDirect P2P 和 SHM 两种方式。

*   当 GPU 之间通过 NVLink 互连时，NCCL 优先使用 GPU 间高速互联的 NVLink 路径实现 GPUDirect P2P 通信
*   当 GPU 之间无 NVLink 互连时，NCCL 还可以使用 GPU 之间的 PCIe 互联总线进行 GPUDirect P2P 通信
*   当 GPU 之间的 P2P 通信不可用或非最优时，NCCL 将使用 SHM 方式，即 GPU-0 的将数据写入主机共享内存，再由 GPU-1 读取

节点间通信
-----

节点间通信，主要有套接字通信，RDMA 通信。

*   当 NIC 不支持 RDMA 时，NCCL 采用套接字进行数据传输。
    *   发送端首先将数据从 GPU 复制到该缓冲区，随后通过标准套接字调用进行网络传输
    *   接收端则将数据接收到主机缓冲区后，再将其复制至 GPU
    *   依赖 Host 内存作为中转的方式，导致额外内存拷贝开销
*   当 NIC 支持 RDMA 时，NCCL 采用 IB 方式进行数据传输。但是分为两种场景：
    *   NIC 无法直接访问 GPU 显存
        *   GPU Kernel 先将数据拷贝至该缓冲区，随后代理线程发起 RDMA 写操作，将数据从 Host 内存传输至远程节点
        *   接收端 NIC 将接收到的数据写入 Host 缓冲区，代理线程协调完成从 Host 内存到 GPU 显存的拷贝
    *   NIC 可以直接访问 GPU 显存
        *   使用条件：仅在 NIC 与 GPU 挂载于同一 PCIe Switch 时启用
        *   中间缓冲区直接分配于 GPU 显存，NIC DMA 引擎可完全绕过 CPU 和 Host 内存，直接对 GPU 执行 RDMA 读写操作
        *   底层通过 IB Verbs 传输，例如：Infiniband（IB），RDMA over Converged Ethernet（RoCE）等

NCCL通信算法简介
==========

NCCL 支持多种集合通信算法，这里主要介绍 ring 和 tree 两种通信拓扑算法。

Ring算法
------

例如对一个浮点数组执行一个 AllReduce Sum 操作，那么预期结果将是每个 GPU 上的浮点数组最后都将是左右 GPU 浮点数组对应元素之和。如下示例所示：

AllReduce 操作将由 Reduce-Scatter + AllGather 共同实现。

*   在 Ring Reduce-Scatter 开始之前，所有 GPU 的 send buffer 包含 k 块独立的数据。Reduce-Scatter 流程总共分为 k - 1 步。在初始阶段，所有 GPU 发送一块本地数据给下一个邻居。在接下来 k − 2 步，所有 GPU 用 recvReduceSend 从上一个邻居接受数据，与本地数据进行求和，并发给下一个邻居；在最后一步，所有 GPU 用 recvReduceCopy 从上一个邻居接受数据，与本地数据进行求和，到此，每个 GPU 都得到了某个对应位置上的元素之和。
*   AllGather 使 k 个 GPU 都获得它们全部的数据。其流程也分为 k - 1 步。在初始阶段，所有 GPU 先将自己的数据拷贝到 output buffer 中，并发送给下一个邻居。若是 in-place 操作，则无需进行拷贝。在接下来 k − 2 步，每个GPU用recvCopySend从上一个邻居接受数据，拷贝到output buffer中，并发送给下一个邻居；在最后一步，每个 GPU 用 recv 接收最后一块数据

因此 ring 算法执行 AllReduce Sum 需要 2k - 2 步数，通信步数为：O(N)

Tree算法
------

如果使用 Tree 算法实现 AllReduce Sum 操作，过程如下图示例：

首先，所有叶子 GPU 将自己的数据 send 给父节点。中间的 GPU 用 recvReduceSend 从子节点接收数据，与本地数据进行 reduce sum，并发给父节点。root GPU 用 recvReduceCopySend 得到最终结果，并发给子节点。然后中间 GPU 用 recvCopySend 接收结果并发给自己的子节点。叶子 GPU 用 recv 接收结果。

因此 tree 算法执行 AllReduce Sum 需要 2 × log₂(N) 步，N 为 GPU 的个数。

### Ring和Tree时延对比

在 NCCL 官方公布的 All Reduce 操作中，比较了 Ring 和 Tree 的时延和带宽差异，在大规模集群场景，Tree 的时延是显著低于 Ring 算法的。

NCCL的一些优化
=========

Tree算法优化
--------

NCCL 先有 Single Binary Tree，后于来在 Single Binary Tree 基础上进行了优化，提出了 Double Binary Tree 的概念。再后来又分别提出了 Split Tree 和 Balanced Tree。这里只涉及 Single Binary Tree 和 Double Binary Tree 的对比。

### Single Binary Tree

首先有如下 Single Binary Tree：

Single Binary Tree 上执行 All Reduce 操作，规约阶段：

*   叶子节点 1, 3, 5, 7,......，31 分别向自己的父节点：2, 6, 10, 14, 18, 22, 26, 30 发送数据
*   非叶子节点 2, 6, 10, 14, 18, 22, 26, 30 把从子节点收到的数据与本地数据进行规约，再将规约后的数据分别发送给自己的父节点
*   依此类推，直到 root 节点 0 规约完成所有数据，时间开销 log\_N

Broadcast 阶段：

*   root 节点将规约后的数据广播给自己的子节点 16
*   子节点将收到的数据继续广播给自己的子节点 8，24
*   依次类推，直到所有的子节点都收到 root 节点的规约数据。时间开销 log\_N

**缺点：**

*   reduce 和 broadcast 操作在时间线上只能串行执行，需要 root 完成 reduce 后才能进行 broadcast。时间开销比较大
*   叶子节点只有一个 parent 节点，没有 child 节点；非叶子节点虽然有两个 child 节点，但是也只有一个 parent 节点。这种结构会导致带宽浪费

### Double Binary Tree

有如下双二叉树结构：

Double Binary Tree 与 Single Binary Tree 的区别：

*   Double Binary Tree 基于 Single Binary Tree 的所有节点生成了第二棵单二叉树
*   相对于 Single Binary Tree，在第二棵二叉树中，节点的角色进行了互换，叶子节点变为非叶子节点，非叶子节点转变为叶子节点
*   如此，除了 root 点，任意节点都有了两个 parent 节点，两个 child 节点。root 节点也有了一个 child 和一个 parent

**Double Binary Tree 带来的优势：**

*   与 Single Binary Tree 形成互补 Tree，通信更加均衡。基本上所有节点都有两个 parent 连接，两个 child 连接
*   增加一个通道，数据可以并行发送和接收，通信时延进一步降低

比如执行 All Reduce 操作时，每个节点的数组分别有 2N 个元素，前 N 个元素被映射到 Double Binary Tree 1，剩下的 N 个元素被映射到 Double Binary Tree 2，数据 reduce 操作可以在两棵 Tree 上并行执行，broadcast 阶段也是如此。

因此 **Double Binary Tree** 的通信时延相对 **Single Binary Tree** 会进一步降低。

多机通信路径选择优化
----------

NCCL 会避免绕路，选择抄近道。

在下图所示的拓扑中，每个 DGX 系统的 NIC-0 连接到同一叶交换机（L0），NIC-1 连接到同一叶交换机（L1），依此类推。这种设计常称为轨道优化（rail-optimized）网络拓扑。轨道优化拓扑有助于最大化全规约（all-reduce）性能，同时减少网络流之间的干扰，还可通过减少轨道间连接来降低网络成本。

在未优化的路径中，消息需经过三层网络交换机（L0、S1和L3），可能因竞争流量而延迟。

NCCL 利用节点内 GPU 间的 NVIDIA NVSwitch 互联特性，通过 PXN（PCI × NVLink），先将数据移至与目标网卡同轨道的 GPU，再直接发送至目标而无需跨轨道传输，实现了消息聚合与网络流量优化，以最大化有效消息速率和网络带宽。
