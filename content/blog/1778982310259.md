三天用 Claude + Kimi 完成一个微信小程序，完整流程全记录
===================================

网上说 AI 能替代程序员的文章很多，说 AI 怎么用的很少。  
  
这篇是后者。记录我用 Claude Code 和 Kimi，从 0 到上线一个完整微信小程序的真实过程，三天完成 MVP，包含支付、AI识别、消息推送。  
  
\---  
做的是什么  
  
小程序叫「到期管家」，管理家里物品的保质期，核心功能：  
  
\- 拍照 AI 识别生产日期、保质期  
\- 条码扫描自动带出商品信息  
\- 到期前微信消息推送提醒  
\- 会员订阅支付体系  

  
技术栈：Taro + React + 微信云开发 + Skyline 渲染框架。  
  
\---  
两个工具怎么分工  
  
先说结论，避免踩我踩过的坑：  
  
Claude Code：主力编码  
Kimi：文档理解 + 长上下文处理  
  
不要用错场景，否则效率反而更低。  
  
\---  
Claude Code 怎么用  
  
全程主力，从组件实现到逻辑重构都靠它。  
  
上下文理解是最明显的优势。把项目结构给它，描述需求，它能准确定位该改哪个文件、影响哪些地方。不会出现"给了一段代码、改了另一段代码"的  
情况。  
  
举一个具体例子。  
  
微信订阅消息有一个隐性约束：requestSubscribeMessage 必须在用户点击事件的同步上下文里调用，放在任何 await  
之后都会静默失败，弹窗不出现，也不报错。这个问题文档里写得很隐晦，stackoverflow 上也很少有人提。  
  
把问题描述清楚，Claude Code 直接给出了正确的重构方案——把订阅授权提到所有异步操作之前，问题解决。  
  
收费是唯一缺点，重度使用成本不低，需要自己判断哪些值得问、哪些自己解决。  
  
\---  
Kimi 怎么用  
  
微信官方文档有时候写得很绕，云开发、Skyline、订阅消息三块文档加起来几万字。  
  
把相关章节直接丢给 Kimi，让它提取关键约束和注意事项，比自己逐字读效率高一倍。  
  
长上下文是它的核心优势，几千行代码一次性放进去不丢信息。  
  
代码生成质量不稳定，复杂业务逻辑不要直接用它生成，理解和查找才是正确姿势。  
  
\---  
三天的实际节奏  
  
第一天：框架搭建 + 核心页面  
  
Taro 项目初始化，Skyline 渲染框架配置，首页、添加页、个人页基础结构完成。  
  
Skyline 是微信今年推的新渲染框架，启动速度比旧框架快将近一倍，接近原生 App 体验。坑也更多，文档不完善，遇到问题基本靠 Claude Code  
解决。  
  
第二天：核心功能

拍照识别、条码查询、云数据库读写、微信消息订阅授权。  
  
条码查询 API 要收费，2 分钱一次，量大了是真实成本。解决方案是把拍照识别做成主路径——对准包装文字拍照，AI  
提取日期信息，类似微信图片编辑里的划选文字交互，成本比条码查询低得多。  
  
第三天：支付 + 收尾  
  
虚拟支付接入，会员体系，边界情况处理，真机测试。  
  
三天 MVP 跑通，等备案的时间顺手把会员订阅和完整支付闭环做完了。  
  
\---  
哪些事 AI 做不到  
  
说完能做的，说不能做的，避免预期偏差。  
  
产品判断做不到。 条码收费这个问题，AI 能帮你实现任何方案，但"用拍照识别代替条码查询"这个决策是自己想的。用错方向，AI  
帮你更快写出一堆没用的代码。  
  
微信平台的隐性规则不熟。 很多约束文档里没写清楚，只能靠踩坑发现。AI 能帮你解决，但得先踩到。  
  
推广做不到。 这是另一个话题，也是比开发难得多的问题。以后单独写。

结论  
  
一个人、三天、完整功能的微信小程序 MVP，在 AI 工具出来之前这件事的成本要高得多。  
  
工具链：Claude Code 写代码，Kimi 读文档，两个配合够用。
